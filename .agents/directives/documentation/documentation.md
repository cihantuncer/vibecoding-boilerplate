# Documentation Directive

Identify audience and durability: user-facing, developer-facing, operational, planning, or code-adjacent. Prefer one source of truth and link to it rather than duplicating volatile details.

Keep docs short, accurate, and structured for scanning. Use examples when they prevent ambiguity. Do not promise behavior that tests or code do not support.

For code comments, explain non-obvious intent, constraints, or risk; do not narrate simple code. For plans and roadmaps, distinguish decisions, options, assumptions, and next steps.

Validate docs against current files, commands, names, and paths. Mention any intentionally omitted detail or stale source risk.
